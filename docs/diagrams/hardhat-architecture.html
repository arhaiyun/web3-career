Hardhat 完整架构与核心认知

以太坊智能合约本地开发环境：以 Hardhat Runtime Environment（HRE）为核心， 串联配置、插件、任务、编译产物与多种网络。本图对应本仓库 projects/hardhat-learning 实践结构。

① 系统上下文 — 谁在用、连到哪里

开发者 编写 Solidity / 测试 / 脚本
CLI npx hardhat <task>
Hardhat 任务调度 · 配置加载 · 插件注入 HRE 核心
区块链网络 内存链 / 本地节点 / 测试网 RPC
开发者 → CLI 命令 → Hardhat 解析 hardhat.config.js → 选定 network → 连接 EVM

② 运行时架构 — HRE（Hardhat Runtime Environment）

Hardhat Runtime Environment
config solidity 版本、networks、paths
tasks compile · test · run · node · clean
plugins toolbox → ethers, chai, coverage…
ethers 合约工厂、部署、调用（经插件）
network 当前连接的 provider 与 accounts
插件在启动时 扩展 tasks 与 HRE 对象；所有脚本通过 require("hardhat") 访问同一 HRE

④ 网络认知 — 三种常用模式

network 场景 特点
hardhat 默认 test / run 内存 EVM，极速，状态可重置
localhost 前端联调、dApp 连接 npx hardhat node（8545）
sepolia 测试网验证 JSON-RPC + 私钥；需测试 ETH

⑤ 核心任务流 — 从源码到链上

compile

  1. 读取 contracts/*.sol
  2. Solidity 编译器（版本与 pragma 一致）
  3. 写入 artifacts/ + cache/

test

  1. 启动 Hardhat Network（内存链）
  2. Mocha 执行 test/*.js
  3. ethers 部署合约并断言

run / node

  1. node：持久本地链 + 预置账户
  2. run scripts/deploy.js：部署到选定网络
  3. 输出合约地址供前端/浏览器使用
典型路径：编写合约 → compile → test →（可选）node + deploy:local →（可选）sepolia 部署

⑥ 部署数据流（Greeter 示例）

Greeter.deploy()
artifacts/Greeter.json

ABI + bytecode（artifacts）+ 签名交易 → 链上合约地址 → 调用 greet() / setGreeting()

⑦ 核心认知卡片 — 学完应能回答

HRE 是什么？

一次命令执行期间的「运行时上下文」：配置、网络、ethers、任务注册表的总线。

插件做什么？

扩展 CLI 任务与 HRE API；toolbox 打包测试、ethers、编译器下载等能力。

artifacts 用途？

含 ABI 与 bytecode；测试/脚本部署时由 Hardhat 自动关联合约名。

内存链 vs 本地节点？

测试用内存链随进程销毁；hardhat node 持久运行，适合 MetaMask 联调。

Gas 从哪来？

内存链/本地链账户自带测试 ETH；测试网需水龙头充值。

config vs .env？

config 写结构与默认值；敏感信息（私钥、RPC URL）放 .env。

compile 与 test 关系？

test 前通常已编译；改 Solidity 后需重新 compile。

安全底线

勿提交私钥；测试网私钥与主网隔离；部署前先在 test 网验证。
